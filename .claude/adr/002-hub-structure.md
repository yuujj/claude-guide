# ADR-002: 허브 구조 — 제품 기준 카드 + 페이지 내부 작업 기준 (하이브리드)

- **상태**: 승인됨
- **일자**: 2026-04-15
- **결정자**: 프로젝트 오너 + AI 어시스턴트
- **관련 문서**: ADR-001 (청중 전략), `.claude/rules/content-plan.md` (v4)

---

## 맥락

기획 v2/v3에서는 허브 카드를 **작업 기준** (대화하기 / 연결하기 / 맡기기)으로 설계했다.
이유: "어떤 도구?"가 아닌 "뭘 하고 싶은가?"로 시작하는 게 UX 원칙에 맞다고 판단.

그러나 샘플 프로토타입(sample.html v3)을 구현·검토한 결과:
- 작업 카드를 본 사용자가 **"그래서 뭘 써야 하는데?"** 라는 기본 질문에 즉답을 못 받음
- "어떤 도구를 쓸까?" 의사결정 테이블로 한 번 더 매핑해야 함 → 두 단계 사고 필요
- v1 샘플에서 쓰던 **제품 카드(웹/데스크탑/코드)** 가 훨씬 직관적이라는 피드백

동시에 v2/v3의 핵심 통찰은 여전히 유효:
- 사용자는 궁극적으로 **자기 목적(task)**을 달성하려 하지, 제품 이름을 알고 싶어하지 않음
- 클로드 코드를 "개발자 전용"으로 닫으면 자동화 관심자를 놓침
- 제품들이 기능을 공유하는 관계(데스크탑 = 웹+MCP+Code)를 기술 기준으로 나누면 혼란

**해결해야 할 질문**: 제품 기준의 직관성과 작업 기준의 깊이를 어떻게 결합할 것인가?

---

## 결정 동인

1. **첫 접점의 명료성** — 허브에서 "뭘 써야 하지?"에 즉답 가능해야 함
2. **페이지 내부 풍부성** — 각 제품 안에서는 "뭘 할 수 있는지"가 더 중요
3. **제품 포함 관계 반영** — 데스크탑이 웹 기능을 포함한다는 현실을 숨기면 안 됨
4. **중간 사용자 배려** — "나는 코딩 몰라도 Claude Code 써볼 수 있나?" 에 열려 있어야 함
5. **유지보수 단일성** — 제품 기능은 변해도 "작업 카테고리"는 상대적으로 안정적

---

## 검토한 옵션

### Option A: 순수 작업 기준 (v2/v3)

- 허브 카드: 대화하기 / 연결하기 / 맡기기
- URL: `chat/`, `connect/`, `automate/`
- 허브에 "어떤 도구를 쓸까?" 매핑 테이블 필요

**장점**: 사용자 목적으로 시작, 기획 원칙 순수 준수
**단점**: "그래서 뭐 써야 해?" 즉답 못 함, 매핑 테이블이 복잡

### Option B: 순수 제품 기준 (v1)

- 허브 카드: 클로드 웹 / 데스크탑 / 코드
- URL: `claude-web/`, `claude-desktop/`, `claude-code/`
- 각 페이지는 제품 기능 나열

**장점**: 직관적 ("웹이네, 눌러야지"), 즉답 가능
**단점**: 사용자 목적을 제품에 맞춰 해석해야 함, Claude Code를 개발자용으로 닫기 쉬움, 기능 나열식이 되기 쉬움

### Option C: 하이브리드 — 제품 기준 허브 + 작업 기준 페이지 내부 (채택)

- **허브**: 제품 카드 3장 (웹/데스크탑/코드) — 직관적 분기
- **허브 보조**: "뭘 하고 싶은지 모르겠다면" 시각적 의사결정 가이드 (3행)
- **각 제품 페이지 안**: "이런 걸 할 수 있어요" 작업별 서브섹션으로 콘텐츠 조직
- **URL**: 제품 기준 (`claude-web/`, `claude-desktop/`, `claude-code/`)

**장점**:
1. 허브에서 "뭘 써야 하지?" 즉답
2. 잘 모르는 사람은 의사결정 가이드로 안내
3. 페이지 안에서는 "뭘 할 수 있는지" 풍부하게 전개
4. 제품 간 포함 관계를 페이지 내부 구조로 자연스럽게 표현 (데스크탑 페이지에서 "웹 기능은 여기서 다 됩니다")
5. Claude Code 페이지 상단에 "이건 코딩 도구인가?" 해명 섹션으로 비개발자 개방 유지

**단점**:
- 기획 원칙 "목적으로 시작"을 엄격히 지키지 못함 (허브는 제품 기준)
- 페이지 내부 콘텐츠 설계 시 "작업 기준"이라는 규칙을 꾸준히 지켜야 함 (문서화 필요)

---

## 결정

**Option C (하이브리드)를 채택한다.**

### 반영 범위

#### 허브 (index.html)
- 섹션 3: "클로드를 쓰는 세 가지 방법" — 제품 카드 3장
- 섹션 4: "뭘 하고 싶은지 모르겠다면" — 시각적 의사결정 가이드 3행 (이모지 + 목적 + 추천 제품 버튼)
- 섹션 5: "가입하고 시작하기" — 3개 시작 방법 탭

#### 제품 페이지 공통 내부 구조
```
[1] Hero (제품 한 줄 정의)
[2] 시작하기 (해당 제품 설치/접속)
[3] 이런 걸 할 수 있어요 ← 작업 기준 서브섹션
[4] 핵심 기능 (아코디언)
[5] 활용 예시 (시나리오)
[6] 요금·한계 or 비교 (제품마다 다름)
[7] 팁
[8] 다음 단계 → 다른 제품 링크
```

#### URL 전환
- `chat/` → `claude-web/`
- `connect/` → `claude-desktop/`
- `automate/` → `claude-code/`
- `tip/` 유지 (ADR-001)

#### 포함 관계 표현
- 데스크탑 페이지 [3] "이런 걸 할 수 있어요" 첫 항목: "웹 기능 전부" → 클로드 웹 페이지로 링크
- Code 탭과 Cowork 탭을 데스크탑 페이지 [5] 핵심 기능으로 다루되, 상세는 각각 클로드 코드 페이지와 Cowork 섹션으로 분기

## 결과

### 양성

- 허브 첫 접점에서 "뭘 써야 하지?" 즉답 가능
- 잘 모르는 사용자도 의사결정 가이드로 3초 안에 추천받음
- 각 제품 페이지 내부에서는 "뭘 할 수 있지?"로 풍부하게 전개 — 기능 나열식 회피
- 제품 포함 관계(웹 ⊂ 데스크탑)가 페이지 내부 구조로 자연스럽게 표현
- Claude Code 페이지 상단 "이건 코딩 도구인가?" 해명으로 비개발자 개방 유지

### 음성

- "목적으로 시작" 원칙을 허브 수준에서는 완전히 관철 못 함 (의사결정 가이드로 보완)
- 페이지 내부 "작업 기준" 구조를 모든 제품 페이지에서 일관되게 유지해야 하는 규율 부담
- ADR-001의 "적응형 컴포넌트" 철학과 결합해서 복잡도 증가 (단, 컴포넌트 자체는 재사용)

### 중립

- URL 명명이 한국어 허브 콘텐츠와 영문 제품명 혼재 (`claude-web/` 등) — 국제 표준과 일치
- v1의 제품 카드 UX와 v3의 적응형 컴포넌트가 결합된 형태 — 각 ADR의 장점을 모두 취함
