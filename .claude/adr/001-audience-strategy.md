# ADR-001: 청중 분기 전략 — 이진 선택 대신 적응형 컴포넌트

- **상태**: 승인됨
- **일자**: 2026-04-15
- **결정자**: 프로젝트 오너 + AI 어시스턴트
- **관련 문서**: `.claude/rules/content-plan.md` (v3)

---

## 맥락

클로드 가이드는 비전공자부터 개발자까지 포괄하는 한국어 정적 가이드 사이트다.
두 청중의 요구가 구조적으로 충돌한다:

| 비개발자가 이탈하는 순간 | 개발자가 이탈하는 순간 |
|---|---|
| 코드 블록이 보임 → "이거 개발자 사이트 아니야?" | USB 비유가 나옴 → "나 개발자인데, 프로토콜 스펙은?" |
| MCP, CLI 같은 용어 → 자신감 하락 | 기술 상세가 disclosure 뒤에 숨어있음 → 번거로움 |
| 선택지가 많음 → 결정 포기 | 예시가 hello world 수준 → 가치 의문 |
| 설치 과정에 터미널 명령 → "나는 아닌가보다" | 빠른 참조 불가 → 재방문 가치 없음 |

**해결해야 할 질문**: 이 충돌을 "사전에 청중을 나누는 것"으로 풀 것인가, "콘텐츠 자체가 적응하게 만들 것"인가?

---

## 결정 동인

1. **유지보수 부담** — 정적 HTML 사이트에서 콘텐츠 이중화는 파일 크기와 동기화 비용을 배로 늘림
2. **중간 사용자 문제** — "비개발자/개발자" 이진 분류에 맞지 않는 사용자가 다수 (CSS 아는 디자이너, 자동화 배우는 PM, 탐색 중인 학생)
3. **확장성** — Phase 3에서 팁 페이지가 추가될 때 구조가 버텨야 함
4. **진입장벽** — 첫 방문자에게 "당신은 누구입니까?" 선택을 강요하면 인지 부하 증가
5. **작업 기준 정합성** — 사이트 구조가 이미 "뭘 하고 싶은가?"(task) 기준이므로 청중 전략도 이에 정합해야 함

---

## 검토한 옵션

### Option A: 청중 기반 스위칭

**메커니즘**: 첫 방문 시 "비개발자/개발자" 선택 → localStorage 저장 → `data-audience` 속성으로 CSS 분기 → 섹션별 show/hide

#### 대상자별 경험

| 대상자 | 경험 |
|--------|------|
| **비개발자** | 코드 블록 숨겨짐, 쉬운 용어만 노출, 시각 가이드 중심 |
| **개발자** | 코드 예시 즉시 노출, 기술 상세 바로 접근, 비유 최소화 |
| **중간 사용자** | 한 쪽을 선택해야 함 — 다른 쪽 콘텐츠를 놓침 |

#### 기능 분석

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | 중 — CSS `display:none` + localStorage + 토글 UI |
| 콘텐츠 유지보수 | **높음** — 4페이지 x ~7섹션 = 28개 콘텐츠 블록이 2벌씩 필요 |
| 첫 방문 경험 | 모달/오버레이로 선택 강요 — 인지 부하 |
| 재방문 경험 | localStorage로 자동 적용 — 빠름 |
| SEO | **불리** — 숨긴 콘텐츠가 중복으로 인식될 위험 |
| Phase 3 확장 | 신규 페이지마다 2벌 작성 필요 — 선형 비용 증가 |

#### UX/UI 요소

- 청중 선택 모달 또는 카드 필요 (첫 방문 시)
- 페이지 어딘가에 "전환" 토글 필요 (잘못 선택 시 복구)
- 모바일에서 선택 모달이 화면을 가림
- 다크모드 전환과 유사한 `[data-audience]` CSS 패턴 — 기술적으로 기존 패턴 재활용 가능

#### 케이스별 장단점

| 케이스 | 장점 | 단점 |
|--------|------|------|
| CSS 아는 디자이너가 Figma MCP 찾을 때 | - | "일반" 선택 시 MCP CLI 설정 못 봄, "개발자" 선택 시 불필요한 git/CI 내용 범람 |
| 시니어 개발자가 JSON config 빠르게 찾을 때 | "개발자" 선택 후 바로 기술 내용 접근 | 선택 단계가 한 번 더 있음 (사소하지만 반복 방문 시 짜증) |
| 학생이 클로드 전체를 탐험할 때 | - | 한쪽만 보게 됨 → 전체 그림 못 그림 (전환 토글 찾아야 함) |
| 회사에서 팀 도입 검토할 때 | - | 공유 URL이 audience 상태를 포함하지 않으면 받는 사람은 기본값으로 봄 |

### Option B: 순수 기능/작업 기준 (현행 v2)

**메커니즘**: 작업 기준 페이지 (대화하기/연결하기/맡기기), 모든 사용자가 동일한 페이지를 봄, progressive disclosure로 깊이 조절

#### 대상자별 경험

| 대상자 | 경험 |
|--------|------|
| **비개발자** | 기본 노출 내용은 이해 가능, 기술 용어 만나면 막힘 |
| **개발자** | 기본 내용이 너무 쉬움, disclosure 여러 번 열어야 원하는 정보 도달 |
| **중간 사용자** | 자연스럽게 자기 수준에서 탐색 |

#### 기능 분석

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | **낮음** — 추가 JS 모듈 불필요 (audience.js 제거 가능) |
| 콘텐츠 유지보수 | **낮음** — 단일 소스, 중복 없음 |
| 첫 방문 경험 | 즉시 콘텐츠 진입 — 장벽 없음 |
| 재방문 경험 | 특별한 개인화 없음 |
| SEO | **유리** — 단일 정본, 모든 콘텐츠 색인 |
| Phase 3 확장 | 1벌만 작성 — 비용 일정 |

#### UX/UI 요소

- 추가 UI 컴포넌트 불필요
- 모든 깊이가 disclosure 뒤에 있어 클릭 비용 누적
- 기술 용어에 대한 브릿지 없음 (비개발자 진입장벽)
- 사이드바 ToC에 깊이 표시 없음 (개발자 탐색 비효율)

#### 케이스별 장단점

| 케이스 | 장점 | 단점 |
|--------|------|------|
| CSS 아는 디자이너가 Figma MCP 찾을 때 | 모든 콘텐츠 접근 가능, 자기 페이스로 탐색 | "MCP" 용어에 설명이 없으면 막힘 |
| 시니어 개발자가 JSON config 빠르게 찾을 때 | 콘텐츠가 있긴 함 | disclosure 여러 개 열어봐야 위치 파악 → 비효율 |
| 비개발자가 "맡기기" 페이지 처음 방문 | 비개발자 예시가 먼저 나옴 (기획대로) | "셸 명령", "git", "CLI" 용어가 그대로 노출 → 위축 |
| 팀 도입 검토 | URL 공유 시 모두 같은 것을 봄 | - |

### Option C: 하이브리드 — 작업 기준 + 적응형 컴포넌트

**메커니즘**: Option B의 구조를 유지하면서, 3가지 적응형 컴포넌트를 추가:

1. **용어 툴팁** — 기술 용어에 점선 밑줄, 클릭/호버 시 팝오버 정의
2. **깊이 배지** — 사이드바 ToC와 disclosure 트리거에 `[고급]` / `[기초]` 배지
3. **TIP 페이지** — 일러스트 포함 독립 페이지, 카테고리별 정리

#### 대상자별 경험

| 대상자 | 경험 |
|--------|------|
| **비개발자** | 기술 용어에 점선 밑줄 → 탭하면 정의 + "TIP에서 자세히" 링크. `[고급]` 배지 보면 "건너뛰어도 됨" 인식 |
| **개발자** | 용어 툴팁은 자연스럽게 무시 (이미 아는 용어). `[고급]` 배지가 사이드바 ToC에서 비콘 역할 → 바로 점프 |
| **중간 사용자** | 필요한 용어만 툴팁으로 확인, 관심 있는 고급 섹션만 선택적 진입 |

#### 기능 분석

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | 중 — 신규 컴포넌트 2개 (term.css/js, toc.css/js) + TIP 페이지 1개 |
| 콘텐츠 유지보수 | **낮음** — 단일 소스 유지. 용어 정의만 별도 관리 |
| 첫 방문 경험 | 즉시 진입 + 용어 장벽 해소 — **최적** |
| 재방문 경험 | 사이드바 ToC + 배지로 빠른 탐색 — **최적** |
| SEO | **유리** — 단일 정본 + TIP이 추가 인덱싱 포인트 |
| Phase 3 확장 | 동일 패턴 재사용, TIP에 항목 추가만 |

#### UX/UI 요소

- **용어 툴팁**: `border-bottom: 1.5px dotted var(--color-accent)`, `cursor: help`
  - 팝오버: card 배경 + shadow-md, 정의 1줄 + TIP 링크
  - 모바일: 탭 → 표시, 바깥 탭 → 닫기 (hover 의존 없음)
  - 접근성: `aria-describedby` 연결
  - `prefers-reduced-motion` 대응: fade 대신 즉시 표시
- **사이드바 ToC**: 데스크탑 sticky, 모바일은 sticky 드롭다운
  - IntersectionObserver로 현재 섹션 하이라이트
  - `[고급]` 배지는 기존 `badge--dev` (인디고) 재사용
- **TIP 페이지**: 카테고리별 정리, 클라이언트 사이드 필터, 각 항목에 일러스트/비유 + disclosure 기술 상세
- **disclosure 배지**: 기존 disclosure 트리거 텍스트에 인라인 배지 추가 — 레이아웃 변경 0

#### 케이스별 장단점

| 케이스 | 장점 | 단점 |
|--------|------|------|
| CSS 아는 디자이너가 Figma MCP 찾을 때 | ToC에서 "Figma MCP" 보고 점프. "MCP" 용어 모르면 툴팁으로 즉시 이해 | - |
| 시니어 개발자가 JSON config 빠르게 찾을 때 | ToC에서 `[고급] MCP 직접 설정` 보고 바로 점프 | 툴팁이 시각적으로 약간 산만할 수 있음 (하지만 무시 가능) |
| 비개발자가 "맡기기" 페이지 처음 방문 | "CLI" 용어에 툴팁 → "명령줄 도구" 정의 표시. `[고급]` 섹션 건너뜀 | 용어 첫 등장 시에만 툴팁 → 이후 등장에서는 직접 TIP 참조 필요 |
| 학생이 전체 탐험 | 모든 콘텐츠 접근 + 용어 학습 가속 + TIP을 사전처럼 활용 | - |
| 팀 도입 검토 | URL 공유 시 동일 경험. TIP 링크를 별도 공유 가능 | - |

## 결정

**Option C (하이브리드)를 채택한다.**

### 핵심 근거

1. **Option B의 모든 구조적 장점을 유지**하면서 (콘텐츠 단일화, 작업 기준, SEO 유리, 중간 사용자 지원),
2. **Option B의 두 가지 결함을 정조준으로 해결**한다:
   - 비개발자의 용어 장벽 → **용어 툴팁 + TIP**
   - 개발자의 깊이 탐색 비효율 → **사이드바 ToC 배지**
3. **Option A 대비 비용이 낮다**: 콘텐츠 이중화(28+블록 x 2벌) 대신 컴포넌트 2개 + 페이지 1개
4. **"중간 사용자 테스트"에서 유일하게 모든 페르소나를 커버**한다

## 결과

### 양성

- 비개발자 이탈률 감소 (용어 장벽 해소)
- 개발자 탐색 효율 향상 (ToC 배지로 빠른 점프)
- 콘텐츠 단일 소스 유지 (유지보수 비용 최소)
- TIP 페이지가 독립적 가치 보유 (검색 유입, 별도 공유 가능)
- 중간 사용자(디자이너, PM, 학생)도 자연스러운 경험

### 음성

- 신규 컴포넌트 2개 + 페이지 1개의 구현 비용 (단, audience 스위칭보다 낮음)
- 콘텐츠 작성 시 `<span class="term">` 마크업 규율 필요
- 용어 정의 동기화 관리 필요 (인라인 data-def ↔ TIP 페이지)

### 중립

- `card--audience` CSS 컴포넌트 제거 — 이미 작성된 코드이나 미사용 상태
- localStorage 기반 개인화 미도입 — 향후 필요 시 추가 가능 (비파괴적)
